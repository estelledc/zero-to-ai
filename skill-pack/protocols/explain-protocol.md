# 六步讲解协议

解释新概念时，遵循以下六步结构。目标是确保用户**真正理解**，而不只是记住。

---

## 六步流程

### 1. 定位

一句话说清这个概念在知识体系中的位置，关联用户已学过的内容。

示例："Redis 持久化是 Redis 数据安全的核心机制，跟你之前学的'内存 vs 磁盘'直接相关。"

### 2. 类比

用日常生活场景建立直觉。必须明确类比的边界（哪里像、哪里不像）。

示例："RDB 像定时拍照片——每隔一段时间把整个房间拍一张全景照。AOF 像记流水账——每做一件事就写一笔。照片恢复快但可能漏掉两次拍照之间的变化；流水账完整但恢复时要从头重放。"

### 3. 机制

从输入到输出的数据/控制流。关键 trade-off 要说清楚。

不需要面面俱到，聚焦"理解这个概念最需要知道的 2-3 个要点"。

### 4. 代码

最小可运行示例（≤20 行）。关键行加注释。

```python
# 示例：Redis RDB 配置
# redis.conf 中的关键配置
save 900 1      # 900秒内有1次写入就触发快照
save 300 10     # 300秒内有10次写入就触发快照
```

如果概念不涉及代码（如设计模式、架构思想），可以用伪代码或配置示例替代。

### 5. 误区

2-3 个初学者常犯的错误。格式：

- 错误认知 → 正确理解

示例：

- "RDB 会丢数据所以不安全" → RDB 和 AOF 可以同时开启，互补使用
- "AOF 文件会无限增大" → Redis 有 AOF rewrite 机制自动压缩

### 6. 验证

2-3 个苏格拉底式应用题。不是背诵题，是需要理解才能回答的场景题。

示例：

- "如果你的应用对数据丢失零容忍，但又要求重启速度快，你会怎么配置？"
- "AOF 的 appendfsync 有三个选项（always/everysec/no），各适合什么场景？"

---

## 使用规则

- **不必每次都走完六步**：用户有基础时可跳过定位和类比，直接从机制开始
- **精读模式**：每步暂停等用户消化，确认理解后再继续
- **快速模式**：一次性给出完整讲解（用户明确要求时）
- **验证必须做**：即使跳过前面的步骤，最后的验证问题不能省略——这是确认理解的关键

## 与学习笔记的映射

六步讲解的内容直接对应 `templates/learning.md` 的结构：

| 讲解步骤    | 笔记段落                  |
| ----------- | ------------------------- |
| 定位        | "是什么"                  |
| 类比 + 机制 | "为什么重要" + "核心要点" |
| 代码        | "代码示例"                |
| 误区        | "常见误区"                |
| 验证        | "自测问题"                |

讲解完成后，可以直接将内容结构化为学习笔记。
