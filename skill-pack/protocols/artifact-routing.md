# 工件选择规则

当用户表达了学习/工作内容时，按以下规则判断记录到哪里。**命中即停**，不重复记录。

---

## 判断矩阵

| 优先级 | 条件                                   | 目标目录     | 使用模板                |
| ------ | -------------------------------------- | ------------ | ----------------------- |
| 1      | 新概念/模式/技能（未来可复用的知识）   | `learnings/` | `templates/learning.md` |
| 2      | 排查 > 30min 且根因清晰可写            | `problems/`  | `templates/problem.md`  |
| 3      | 新的学习材料/资源/文章/课程            | `sources/`   | `templates/source.md`   |
| 4      | 其他（含小困难、todo、心得、日常记录） | `daily/`     | `templates/daily.md`    |

---

## 信号识别

### → learnings/

用户说了类似这些话：

- "我学会了 X" / "搞懂了 X" / "原来 X 是这样"
- "X 的原理是..." / "X 和 Y 的区别是..."
- 对话中解释了一个概念超过 5 句话
- 用户表达了"顿悟"时刻

### → problems/

用户说了类似这些话：

- "终于解决了" / "搞定了那个 bug"
- "花了很久才找到原因"
- "排查了半天发现是..."
- 描述了 现象→排查→根因→解决 的完整过程

### → sources/

用户说了类似这些话：

- "发现一个好资料" / "这篇文章不错"
- "推荐看看这个" / "这个教程很好"
- 分享了 URL 或书名

### → daily/

以上都不命中时，默认记到当天日报：

- "今天做了 X" / "明天要做 Y"
- 小困难（不值得单独开 problem）
- 心得感悟
- 工作进展

---

## 边界情况

**一次对话中同时触发多个**：分别创建。比如用户说"今天解决了一个 bug，过程中学会了 Redis 的过期策略"——创建一个 problem + 一个 learning。

**不确定是 learning 还是 daily**：问用户"这个知识以后还会用到吗？如果会，我帮你单独记一篇；如果只是今天的小发现，记到日报里。"

**已有相关笔记**：创建前先 grep `learnings/` 目录。如果已有相关笔记，建议补充而不是新建。

---

## 命名规范

| 目录         | 命名格式        | 示例                      |
| ------------ | --------------- | ------------------------- |
| `daily/`     | `YYYY-MM-DD.md` | `2026-06-17.md`           |
| `learnings/` | `kebab-case.md` | `redis-persistence.md`    |
| `problems/`  | `kebab-case.md` | `docker-port-mapping.md`  |
| `sources/`   | `kebab-case.md` | `redis-in-action-book.md` |
